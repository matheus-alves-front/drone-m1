# Mark 2 Roadmap

## P0 — Freeze Mark 1 Compatibility
- garantir que o Mark 1 final respeitou todas as regras de compatibilidade
- registrar dívidas assumidas conscientemente

## P1 — Capability Registry Runtime
- criar registry real
- capability discovery dinâmico
- versionamento de capability

## P2 — Payload Runtime
- modelar payloads formais
- mount/unmount
- health/status
- binding com capability registry

## P3 — Actuator Runtime
- modelar actuators
- comandos
- safety envelopes
- read model específico

## P4 — Vehicle Family Generalization
- separar drone como uma família de veículo
- abrir espaço para futuras bases móveis ou células robóticas

## P5 — MCP Surface
- expor tools e contracts para IA
- validar boundary safety-aware

## P6 — Hardware Path
- definir trilha realista de hardware adapters
- definir readiness checklist de V1 física futura

## Regra
Nenhum desses blocos deve puxar implementação prematura para o Mark 1 fora do necessário para compatibilidade.
